## What Failed / Issues Found

1. **Bootstrap git safety paradox.** The implementer violated the git safety rule it was establishing — task-07 and code-review-fixes-r1 both log `git add -A` in commands_executed. The rule was active (task-01 completed 40 minutes earlier) but the implementing agent still staged files.

2. **Audit pattern alignment gap.** Tester and reviewer audit patterns use narrower patterns than the implementer's actual allowlist. Three legitimate commands (`dotnet run`, `npm test`, `python -m pytest`) would generate false-positive violations.

3. **Line count reporting drift.** Implementation reports self-report inaccurate line counts for 2 files (planner: claimed 120, actual 112; knowledge: claimed 117, actual 118).

4. **global-rules.md agent detection.** VS Code auto-detects any .md file in .github/agents/ as an agent. global-rules.md lacks YAML frontmatter and appears as an unintended agent.

---

## Root Causes

- **Bootstrap paradox:** The pipeline was modifying the v2 agent files that define the rules the pipeline should follow. Git safety could not be enforced before it was implemented. Future self-improvement runs need orchestrator-only staging from task-01 onward.

- **Audit pattern shorthand:** The tester/reviewer patterns used pipe-delimited shorthand (`dotnet build|test`) derived from the design, but the implementer's allowlist was expanded during implementation to include `dotnet run`. The patterns diverged during the design→implementation gap.

- **Line count estimation:** Implementers estimate line counts in their reports rather than capturing exact terminal output. Multi-task modifications to the same file (e.g., orchestrator modified in task-02 and wave-by-wave additions) create measurement timing issues.

## Improvement Suggestions

1. **Enforce git safety from task-01.** The orchestrator should assume staging responsibility immediately after the git safety rule is established. Implementers should report `git_staged: false` and the orchestrator runs `git add` with pathspecs from reports.

2. **Align audit patterns in design.** The designer should produce a canonical command allowlist that tester/reviewer/implementer all reference verbatim, rather than independently summarizing.

3. **Capture line counts from terminal.** Add a verification step where implementers run `(Get-Content file).Count` and copy the exact number into the report, eliminating estimation.

4. **Add frontmatter to global-rules.md.** Minimal `user-invocable: false` frontmatter prevents VS Code from showing it as an agent.

5. **Expand E2E policy.** User requested: 🟡 risk tasks should also require E2E testing when an e2e-contract.yaml is present (currently only 🔴 is mandatory).
