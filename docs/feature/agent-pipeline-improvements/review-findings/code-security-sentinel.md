### Finding S-1: Fast-track prompt skips design review (Step 3b) — no adversarial security review before implementation

- **Severity:** Critical
- **Description:** The `plan-and-implement.prompt.md` fast-track pipeline defines a reduced step set {0, 4, 5, 6, 7, 8, 9} that skips Steps 1–3b (research, spec, design, **design review**). While the anti-drift anchor mandates minimum steps {0, 7, 9}, the design review at Step 3b is a critical security checkpoint that catches architectural vulnerabilities _before_ implementation begins. The fast-track prompt's "Scope Guardrails" section says it's "for small, low-risk (🟢) features only" — but this is advisory text, not an enforcement gate. Nothing in the orchestrator prevents a user from invoking the fast-track prompt for 🔴-risk work, bypassing the design review entirely.
- **Affected artifacts:** [plan-and-implement.prompt.md](NewAgents/.github/prompts/plan-and-implement.prompt.md), [orchestrator.agent.md](NewAgents/.github/agents/orchestrator.agent.md#L120) (Step 0 item 6.a)
- **Recommendation:** Add enforcement in the orchestrator: when `pipeline_mode=fast-track` and the planner classifies any task as 🔴 risk, the orchestrator MUST either (a) escalate to full pipeline mode automatically, or (b) halt with an error requiring the user to use `feature-workflow.prompt.md`. The scope guardrails in the prompt itself are insufficient — they are suggestions, not gates. Additionally, consider adding Step 3b (code review exists at Step 7, but design review at 3b catches architectural flaws earlier).
- **Evidence:** `plan-and-implement.prompt.md` lines 18–22 contain advisory "Scope Guardrails" with no enforcement mechanism. Orchestrator line 120: `6.a **Pipeline mode (D-10):** If invoked via plan-and-implement.prompt.md, set pipeline_mode: fast-track (skip Steps 1–3b).` — no risk-level guard.

### Finding S-2: `fetch_webpage` restriction is instruction-based only — no runtime enforcement prevents autonomous-mode invocation

- **Severity:** Major
- **Description:** The `fetch_webpage` tool is restricted to interactive mode via instructional text in tool-access-matrix.md §3/§4/§5 and individual agent files. However, the spec-output.yaml constraint section explicitly states: "Tool access enforcement is instruction-based only (no runtime enforcement). See tool-access-matrix.md §11." This means there is no runtime mechanism to prevent an agent running in autonomous mode from calling `fetch_webpage`. The agent would need to self-enforce. While VS Code's `fetch_webpage` does require user interaction (approval per invocation), other tool environments may not have this gating. The lack of runtime enforcement is a known constraint — but the multi-layered instructional defense is adequate given that constraint.
- **Affected artifacts:** [tool-access-matrix.md](NewAgents/.github/agents/tool-access-matrix.md) §3, §4, §5; [researcher.agent.md](NewAgents/.github/agents/researcher.agent.md) lines 136–148; [spec.agent.md](NewAgents/.github/agents/spec.agent.md) lines 242–250; [designer.agent.md](NewAgents/.github/agents/designer.agent.md) lines 162–179
- **Recommendation:** The instructional defense is well-implemented with 4 reinforcement layers (master matrix, per-agent scope section, agent file workflow step, agent file anti-drift anchor). Accept the instruction-based enforcement as the best available mechanism, but add a self-verification checklist item to each agent with `fetch_webpage` access: "[ ] `fetch_webpage` NOT used when `approval_mode='autonomous'`". This creates an auditable trail even under instruction-only enforcement.
- **Evidence:** spec-output.yaml constraints: "Tool access enforcement is instruction-based only (no runtime enforcement)." tool-access-matrix.md §3: "Agents in autonomous mode MUST NOT attempt to use fetch_webpage." Researcher, spec, designer agents each have Skip conditions gated on `approval_mode`. No self-verification checklist item for this constraint exists in any of the three agents.

### Finding S-3: DB archive rename expands orchestrator filesystem scope — potential path injection via crafted `run_id` timestamp

- **Severity:** Major
- **Description:** The DB archive feature (D-9) allows the orchestrator to perform filesystem rename operations (`Rename-Item`/`mv`) for the first time. The rename target is `verification-ledger-{timestamp}.db`. The `{timestamp}` is the ISO 8601 `run_id` generated by the orchestrator itself at Step 0. While the orchestrator generates this value, a malicious or manipulated `initial-request.md` could attempt to influence the `run_id` value. More importantly, tool-access-matrix.md §2 constrains the rename to "ONLY for renaming `verification-ledger.db` to `verification-ledger-{timestamp}.db`" — but this is again instructional, not enforced by a regex or allowlist pattern. The `run_id` value typically follows ISO 8601 (`2026-03-05T12:00:00Z`), which contains characters safe for filenames (digits, hyphens, colons, T, Z). On Windows, `:` is invalid in filenames and may cause errors or unexpected behavior.
- **Affected artifacts:** [orchestrator.agent.md](NewAgents/.github/agents/orchestrator.agent.md#L111) line 111 (Step 0 3.a); [tool-access-matrix.md](NewAgents/.github/agents/tool-access-matrix.md) §2; [sql-templates.md](NewAgents/.github/agents/sql-templates.md) §1.1
- **Recommendation:** (1) Document in sql-templates.md §1.1 that the timestamp in the filename MUST use filesystem-safe format (replace `:` with `-` or use compact format like `20260305T120000Z`). ISO 8601 with colons is invalid on Windows NTFS. (2) The archive rename scope in tool-access-matrix.md §2 is appropriately narrow, but explicitly state that the source must be exactly `verification-ledger.db` and the target must match the pattern `verification-ledger-[\w\-]+\.db$` or `verification-ledger-[\w\-]+-corrupt\.db$`.
- **Evidence:** orchestrator.agent.md line 111: `rename to verification-ledger-{timestamp}.db`. sql-templates.md §1.1 step 2: `rename the DB to verification-ledger-{timestamp}.db`. ISO 8601 timestamps contain `:` which is forbidden in Windows NTFS filenames (e.g., `verification-ledger-2026-03-05T12:00:00Z.db` is invalid on Windows).
